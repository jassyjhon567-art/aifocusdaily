HollowByte Vulnerability in OpenSSL Allows Tiny 11-Byte Requests to Freeze Server Memory

thehackernews.com · Sat, 18 Jul 2026 01:50:53 +0530

The HollowByte vulnerability in OpenSSL allows attackers to freeze server memory using tiny 11-byte TLS requests, leading to potential denial-of-service conditions on unpatched systems.

A newly disclosed vulnerability in the widely used OpenSSL cryptographic library, dubbed "HollowByte," could allow remote attackers to freeze server memory using extremely small network requests. Security researchers revealed that an unpatched OpenSSL server can be triggered to set aside up to 131 kilobytes of buffer memory by sending a specially crafted TLS request of just 11 bytes. Because the expected handshake message never actually arrives, the server keeps this memory reserved indefinitely, paving the way for devastating denial-of-service (DoS) attacks.

The real-world impact of this flaw is particularly severe on systems using the standard GNU C Library (glibc). According to extensive testing conducted by Okta's Red Team, the memory allocated during these malicious, incomplete requests is effectively lost and cannot be reclaimed by the operating system until the affected OpenSSL server process is completely restarted. By repeatedly sending these tiny 11-byte packets, an attacker could rapidly exhaust a target server's available memory resources with minimal network bandwidth and effort.

Although OpenSSL quietly shipped a software fix for the HollowByte bug in June, the patch was deployed without an accompanying Common Vulnerabilities and Exposures (CVE) identifier, an official security advisory, or any explicit mention in the release's public changelog. This lack of standard public documentation left many system administrators completely unaware of the underlying security risk. Okta's Red Team, which initially discovered, reported, and named the vulnerability, has now published its detailed research to encourage organizations to update their OpenSSL deployments immediately.
